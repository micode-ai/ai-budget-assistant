# AI Budget Assistant

## Architecture

Turborepo monorepo with 5 packages:

| Package | Tech | Purpose |
|---|---|---|
| `apps/api` | NestJS 10 + Prisma 5 + PostgreSQL + Redis | REST API backend |
| `apps/mobile` | Expo 54 + React Native 0.81 + Zustand + SQLite/Drizzle | Mobile app (iOS/Android/Web) |
| `apps/admin` | Next.js 16 + React 19 + TailwindCSS 4 + shadcn/ui + Recharts | Admin web dashboard (port 3001) |
| `packages/shared-types` | TypeScript interfaces | Entities and DTOs shared between api and mobile |
| `packages/shared-utils` | Zod schemas + formatting + constants | Validation and utilities shared between api and mobile |

## Key Patterns

### API (NestJS)
- **Module structure**: `modules/<feature>/` contains `module.ts`, `controller.ts`, `service.ts`, `dto/index.ts`, `guards/`
- **Auth**: JWT via `@UseGuards(JwtAuthGuard)`. Request type: `AuthenticatedRequest` from `common/types/index.ts`. Password reset via 6-digit email code (`POST /auth/forgot-password`, `POST /auth/reset-password`) with in-memory rate limiting and bcrypt-hashed codes. Email change also via 6-digit code (`POST /auth/change-email/request`, `POST /auth/change-email/confirm`) — JWT-guarded, requires current password on request, issues new tokens on confirm. **Google sign-in (ABA-282)**: `POST /auth/google` (public) takes a Google **ID token** from the client (mobile + web via `expo-auth-session`, no native module), verified server-side by `GoogleTokenVerifier` (`google-auth-library`, audiences from `GOOGLE_OAUTH_CLIENT_IDS`, requires `email_verified`). Resolves by `googleId` → **auto-link by verified email** (sets `googleId`; rejects + does not link a deactivated account) → else creates a verified, passwordless user (name/email from Google, language from client, currency default USD) + default account. Returns the same shape as `/auth/login`. `User.googleId String? @unique` + `passwordHash` now **nullable** (migration `20260621000000_add_google_auth`); `login()` and `changeEmailRequest()` reject null-passwordHash accounts ("use Google sign-in"). Mobile: `authStore.googleLogin`, `useGoogleAuth` hook, "Continue with Google" on `(auth)/login.tsx`+`register.tsx`. **Web nonce (ABA-291)**: the web path uses the **generic** `expo-auth-session` `useAuthRequest` with `responseType: IdToken`, which (unlike `providers/Google`) does NOT add a `nonce` — Google's OpenID implicit flow requires one, so without it web sign-in failed with `Error 400: invalid_request / GeneralOAuthFlow` (native was fine — `signInNative()` builds its own URL with a manual nonce). Fix: `useGoogleAuth` generates a `nonce` once (`useMemo(() => rnd(), [])`) and passes `extraParams: { nonce }` to the web `useAuthRequest`. **Prerequisite**: Google Cloud OAuth client IDs (`GOOGLE_OAUTH_CLIENT_IDS` on API; `EXPO_PUBLIC_GOOGLE_WEB/IOS/ANDROID_CLIENT_ID` on client)
- **Account scoping**: `AccountContextGuard` middleware reads `X-Account-Id` header, resolves membership, adds `accountId` and `accountRole` to request
- **Role-based access**: `AccountRoleGuard` with `@RequireRole('owner'|'editor')` decorator (DI-based, needs `AccountsModule`). `ViewerBlockGuard` (ABA-155) — zero-dependency guard in same file; use as `@UseGuards(new ViewerBlockGuard())` on any write method without importing `AccountsModule`. Reads `req.accountRole` set by `AccountContextGuard`. Apply to all POST/PATCH/PUT/DELETE endpoints that mutate account-scoped data. AI chat: viewer write-actions blocked in `chat.service.ts` before a pending action is queued. Bots: `TelegramUserState` and `WhatsAppUserState` include `accountRole`; link services join `AccountMember` on `getLink()`; write handlers check the role before executing.
- **Service signature**: `(accountId, userId, dto)` as parameters, all Prisma queries filter by `accountId`
- **Transaction attribution**: `Expense.userId` and `Income.userId` already store the creator's id (set from `request.userId` at create time). Both services include `user: { select: { name: true } }` in all Prisma queries and call a `toExpenseResponse()`/`toIncomeResponse()` helper to flatten it into `createdByUserName: string | null` on the returned object. No migration needed — just always pass the user include and map the result.
- **Database**: Prisma ORM. Schema at `apps/api/prisma/schema.prisma`. Uses `@map("snake_case")` for column names. Prod `DATABASE_URL` includes `connection_limit=10` (see `docker-compose.prod.yml`) to cap the connection pool.

- **Purchase requests** (ABA-298): `modules/purchase-requests/` — group purchase approval for shared accounts. Any member (including viewers) can propose a purchase; other members vote via app or bots; approval creates a planned expense. **Data model**: `purchase_requests` + `purchase_request_votes` tables (migrations `20260630000000_add_purchase_requests`); `Account.purchaseApprovalRule ApprovalRule @default(MAJORITY)` (migration `20260630000001_add_account_approval_rule`); `Expense.isPlanned Boolean @default(false)` + `@@index([accountId, isPlanned])` (migration `20260630000002_add_expense_is_planned`). **Approval rules**: `MAJORITY` (approve/reject count > 50% of total members); `UNANIMOUS` (all must approve; one reject kills it); `OWNER_ONLY` (only the account owner's vote counts). ABSTAIN excluded from denominator so non-participating members don't block decisions. Voting is upsert on `[requestId, userId]` so members can change their vote. **9 endpoints** (all `JwtAuthGuard + AccountContextGuard`): `GET/POST /purchase-requests`, `GET/POST /purchase-requests/:id`, `POST /purchase-requests/:id/vote` (any member incl. viewers — intentionally NOT behind `ViewerBlockGuard`), `POST /purchase-requests/:id/convert` (ViewerBlockGuard — creates planned `Expense`, sets `plannedExpenseId`), `POST /purchase-requests/:id/mark-purchased` (ViewerBlockGuard — sets `status = PURCHASED`), `PATCH /purchase-requests/:id` (ABA-302 — edit PENDING request, ViewerBlockGuard, creator or owner only), `DELETE /purchase-requests/:id` (creator or owner only), `PATCH /purchase-requests/settings/approval-rule` (owner only, declared first to avoid `:id` shadow). Route order matters: `settings/approval-rule` → `pending-count` → `/` → `/:id` → `/:id/vote` → `/:id/convert` → `/:id/mark-purchased` → `PATCH /:id` → `DELETE /:id`. **Edit/Delete (ABA-302)**: creator or account owner can edit (title/amount/currency/merchant/description) or delete (= set status REJECTED + fire family-feed rejected event) a PENDING request. `[id].tsx` shows Edit + Delete buttons only for PENDING requests; Edit navigates to `new.tsx?editId=` which prefills fields and calls `updateRequest` on save; viewers cannot edit. **Notifications**: 4 push types — `purchase_request_created` (all members except creator), `purchase_request_voted` (creator), `purchase_request_approved` (all members), `purchase_request_rejected` (creator). User preference `notifyPurchaseRequests Boolean @default(true)` gated in `NotificationsService.sendToUser`. **Telegram bot**: `PurchaseRequestHandler` in `modules/telegram/handlers/purchase-request.handler.ts`, callbacks `pr_approve:{id}` / `pr_reject:{id}` (`:` separator). **WhatsApp bot**: same via `sendButtons`, callbacks `pr_approve--{id}` / `pr_reject--{id}` (`--` separator). **Planned expense**: `loadAllExpenses` in `expenseRepository.ts` filters `AND (is_planned IS NULL OR is_planned = 0)` so planned expenses never inflate spend totals. `expense/[id].tsx` shows a "Planned · Mark as purchased" banner when `isPlanned: true`; tap: optimistic `updateExpense({ isPlanned: false })` FIRST, then `markAsPurchased(linkedPR.id)`, catch → rollback + alert. Mobile: `purchaseRequestStore.ts` (server-only, not offline-first — voting requires cross-member consistency); 3 screens `app/purchase-requests/index.tsx` (3 tabs Active/Approved/History), `new.tsx`, `[id].tsx`; entry via Settings hub row with pending-count badge. 34 `purchaseRequests.*` i18n keys in all 9 locales. Tests: 27/27 (service 15, controller 12). **Phase 2 deferred**: expiry cron, comments thread, Family Feed integration, push image to bots.

## Dependency Order for Changes

When modifying features that span multiple packages, follow this order:

1. `packages/shared-types` — entity interfaces and DTOs
2. `packages/shared-utils` — Zod validation schemas (if needed)

4. `apps/api/src/modules/*` — API services, controllers, guards

## Production

Hetzner VPS (Hetzner cloud), Docker Compose. Stack defined in `docker-compose.prod.yml`:
- `budget-db-prod` (postgres:16-alpine, 512M), `budget-redis-prod` (redis:7-alpine, 320M — `maxmemory 256mb allkeys-lru`), `budget-api-prod` (1G, `NODE_OPTIONS=--max-old-space-size=768`), `budget-admin-prod` (256M), plus shared `accounting-nginx` (separate stack) reverse-proxying `api.ai-budget.pl` and `admin.ai-budget.pl`. API container has **no host port mapping** — reach it only through nginx or the docker network. **API memory (ABA-163)**: without an explicit `NODE_OPTIONS=--max-old-space-size`, Node auto-caps the V8 heap at ~half the container limit (~256M at 512M), which OOM-crashed the backup export. Compose sets heap to 768 and `deploy.resources.limits.memory` to 1G — keep them in sync (heap + ~150M non-heap must fit under the limit).
- Volumes: `ai-budget_postgres_data`, `ai-budget_redis_data` (preserve across deploys; never `docker volume prune` without filters).
- **DB backups (ABA-163, Phase 1)**: `.github/workflows/backup-db.yml` — nightly cron (`0 2 * * *` UTC, off-host so it survives a VPS outage) + `workflow_dispatch`. SSHes to the VPS, runs `pg_dump -Fc` inside `budget-db-prod`, streams it to the runner, validates it (size ≥10 KB + `pg_restore --list` object count), `age`-encrypts (asymmetric — runner holds only `AGE_PUBLIC_KEY`; the private key is **offline**, never in CI), and publishes it as a Release asset (`ai_budget-YYYY-MM-DD.dump.age`, tag `backup-YYYY-MM-DD`) in the private repo `BACKUP_REPO` via `BACKUP_REPO_TOKEN` (PAT, `contents:write`). Logic lives in `scripts/backup-db.sh` (dump+sanity+encrypt; mockable via `LOCAL_DUMP`) and `scripts/prune-backups.sh` (GFS retention 7 daily/4 weekly-Sunday/6 monthly-1st; mockable via `RELEASES_FILE`/`DRY_RUN`). Failure → Telegram alert (same pattern as `uptime-check.yml`). Restore: `docs/ops/restore-runbook.md` (decrypt → scratch-restore → verify → prod `pg_restore --clean --if-exists`). RPO ≤24 h. Redis is **not** backed up (cache/ephemeral). Receipt images are in-DB `Bytes`, so the dump captures them (and grows with them — revisit object storage if a dump exceeds a few hundred MB). Phases 2–4 (infra fault tolerance, HA, mobile local backup) deferred — see `docs/superpowers/specs/2026-05-29-postgres-backup-phase1-design.md`.
- Deploy: push to `development` triggers `.github/workflows/deploy.yml` → SSH → `scripts/deploy.sh` (`git reset --hard`, `npm install` with lock, build, `prisma migrate deploy`, `up -d --force-recreate api admin`, then `docker image prune -f` + `docker builder prune -af`). Verify-step polls `https://api.ai-budget.pl/api/v1/health` for up to 120 s. **Build-cache pruning matters**: images are built on the VPS each deploy and the build cache accumulated ~4 GB → disk hit 89% (ABA-165); the per-deploy `builder prune` keeps it in check. `.github/workflows/infra-diagnostics.yml` (manual `workflow_dispatch`) dumps a `docker system df` + `du` breakdown over SSH when disk needs investigating.
- **Snap-installed Docker is held and disabled** (`snap refresh --hold docker`, `snap disable docker`) after the 2026-04-27 incident where snap auto-refresh hijacked `/var/run/docker.sock` from the apt-installed daemon. Do not re-enable snap docker; only the apt-installed `dockerd` (system data root `/var/lib/docker`) owns prod state.
- Adding env vars: `docker restart` does NOT reload `env_file`. Must `docker compose -f docker-compose.prod.yml --env-file .env.production up -d --force-recreate <service>`.
- **CORS (ABA-151)**: `main.ts` builds `corsOrigins` from `process.env.CORS_ORIGIN.split(',')` (falls back to `localhost:8081`/`:3001` when unset). It must be an **explicit comma-separated origin list** in prod `.env.production` — e.g. `CORS_ORIGIN=https://admin.ai-budget.pl,https://ai-budget.pl`. Do **NOT** use `CORS_ORIGIN=*`: it becomes the array `["*"]`, which Express/Nest `cors` matches as a literal origin (not a wildcard), so the admin browser gets no `Access-Control-Allow-Origin` and login silently breaks (and `*` is invalid with `credentials:true`). Native mobile is unaffected (no `Origin` header). After changing it, force-recreate the `api` service (see env-var note above).

## Mobile releases (Android)

- **Bare Expo workflow** — `apps/mobile/android/` is committed, so EAS builds from the native project and does **not** run prebuild or read `app.json` for the version. The version shown in **Google Play comes from `apps/mobile/android/app/build.gradle` `versionName`**, not `app.json`. `eas.json` `appVersionSource: "remote"` + production `autoIncrement: true` only manage the integer `versionCode` (build number) remotely; they never touch `versionName`. (Historically this caused every release to ship as `1.0.0` while `versionCode` quietly incremented — ABA-131.)
- **Before each release**: bump `versionName` in `android/app/build.gradle` (source of truth for Play) AND `version` in `app.json` (kept in sync for the in-app update gate) to the same value. Do not hand-edit `versionCode` — EAS remote autoIncrement owns it.
- **CI guard**: `apps/mobile/scripts/check-release-version.js` runs as a production-only step in `mobile-build.yml` and `mobile-eas-build.yml`. It fails the build if `build.gradle` versionName ≠ `app.json` version, and (best effort) if the version isn't strictly greater than the latest `v*.*.*` git tag (skipped when no release tags exist).
- **Publish**: run the *Mobile Build & Publish* workflow (`mobile-build.yml`, profile `production`, submit `true`) → builds AAB `--local` → `eas submit` to the Play production track. After release, also add the new version in **admin → App Versions** so the in-app `UpdatePrompt` knows about it.

## Observability

- **Health**: `GET /api/v1/health` (public, no auth) — runs `SELECT 1` on Postgres, returns `{status, db, uptimeSeconds, timestamp}`. 503 if DB fails. Used by Docker `HEALTHCHECK` (requires HTTP 200) and CI verify-step.
- **Uptime**: `.github/workflows/uptime-check.yml` runs every 5 min via cron, hits the public `/api/v1/health`. On non-200, sends a Telegram alert via `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` GitHub Secrets with HTTP code, response body, and a link to the failed run.
- **Backup alerts (ABA-163)**: `backup-db.yml` posts a Telegram alert on any failed nightly DB backup (`TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID`), same channel as the uptime check. A silent backup failure is the classic way to discover at restore time that there are no backups — so failures, not successes, are alerted.
- **Infra watch (ABA-165)**: `.github/workflows/infra-watch.yml` — cron every 30 min, SSHes to the VPS and pipes `scripts/infra-check.sh` (`ssh … 'bash -s' < script`) to check root-disk usage (> 85% alerts) and that all 4 prod containers are running and not `unhealthy`; non-zero exit → Telegram alert. Complements `uptime-check.yml` (5-min `/health`) with disk pressure + per-container granularity. Reuses SSH + Telegram secrets (no new ones).
- **Log rotation (ABA-165)**: `docker-compose.prod.yml` sets `logging` `json-file` `max-size: 10m` / `max-file: 3` on postgres/redis/api/admin (~30 MB/container) so logs can't fill the disk.
- **Disaster recovery**: `docs/ops/restore-runbook.md` (DB-only restore) and `docs/ops/disaster-recovery-runbook.md` (full server rebuild after total VPS loss). Both need the offline `age` private key; full rebuild also needs an **offline copy of `.env.production`** (prod secrets live only on the VPS). Redis is cache-only and intentionally not persisted.
- **Errors**: `@sentry/node` v8 captures unhandled exceptions and Express 5xx errors. Init must remain at the top of `main.ts` via `import './instrument'` — moving it after other imports breaks instrumentation.
- **No Prometheus / Datadog / external APM** at this time.
