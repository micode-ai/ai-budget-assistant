# AI Budget Assistant

## Architecture

Turborepo monorepo with 5 packages:

| Package | Tech | Purpose |
|---|---|---|
| `apps/api` | NestJS 10 + Prisma 5 + PostgreSQL + Redis | REST API backend |
| `apps/mobile` | Expo 54 + React Native 0.81 + Zustand + SQLite/Drizzle | Mobile app (iOS/Android/Web) |
| `apps/admin` | Next.js 16 + React 19 + TailwindCSS 4 + shadcn/ui + Recharts | Admin web dashboard (port 3001) |
| `packages/shared-types` | TypeScript interfaces | Entities and DTOs shared between api and mobile |
| `packages/shared-utils` | Zod schemas + formatting + constants | Validation and utilities shared between api and mobile |

## Key Patterns

### API (NestJS)
- **Module structure**: `modules/<feature>/` contains `module.ts`, `controller.ts`, `service.ts`, `dto/index.ts`, `guards/`
- **Auth**: JWT via `@UseGuards(JwtAuthGuard)`. Request type: `AuthenticatedRequest` from `common/types/index.ts`. Password reset via 6-digit email code (`POST /auth/forgot-password`, `POST /auth/reset-password`) with in-memory rate limiting and bcrypt-hashed codes. Email change also via 6-digit code (`POST /auth/change-email/request`, `POST /auth/change-email/confirm`) â€” JWT-guarded, requires current password on request, issues new tokens on confirm
- **Account scoping**: `AccountContextGuard` middleware reads `X-Account-Id` header, resolves membership, adds `accountId` and `accountRole` to request
- **Role-based access**: `AccountRoleGuard` with `@RequireRole('owner'|'editor')` decorator (DI-based, needs `AccountsModule`). `ViewerBlockGuard` (ABA-155) â€” zero-dependency guard in same file; use as `@UseGuards(new ViewerBlockGuard())` on any write method without importing `AccountsModule`. Reads `req.accountRole` set by `AccountContextGuard`. Apply to all POST/PATCH/PUT/DELETE endpoints that mutate account-scoped data. AI chat: viewer write-actions blocked in `chat.service.ts` before a pending action is queued. Bots: `TelegramUserState` and `WhatsAppUserState` include `accountRole`; link services join `AccountMember` on `getLink()`; write handlers check the role before executing.
- **Service signature**: `(accountId, userId, dto)` as parameters, all Prisma queries filter by `accountId`
- **Transaction attribution**: `Expense.userId` and `Income.userId` already store the creator's id (set from `request.userId` at create time). Both services include `user: { select: { name: true } }` in all Prisma queries and call a `toExpenseResponse()`/`toIncomeResponse()` helper to flatten it into `createdByUserName: string | null` on the returned object. No migration needed â€” just always pass the user include and map the result.
- **Database**: Prisma ORM. Schema at `apps/api/prisma/schema.prisma`. Uses `@map("snake_case")` for column names. Prod `DATABASE_URL` includes `connection_limit=10` (see `docker-compose.prod.yml`) to cap the connection pool.

- **Budget category model**: `Budget` has **no** `categoryId` field. Category filtering is done exclusively through `BudgetCategory` allocations (`budget_categories` table). The legacy `category_id` column was removed via migration `20260523000000_drop_budget_category_id` (backfill + drop). Mobile `budgets` SQLite table still has a dead `category_id` column on existing devices but it is never read or written.

- **Backups** (ABA-163): `modules/backups/` â€” `POST /backups/export` builds a full account snapshot, serializes it **once**, and streams the JSON string straight to the response body via `@Res()` (no Nest re-serialization â€” returning `data: backup` previously tripled peak memory and OOM-crashed the API). Filename travels in the `X-Backup-Filename` header (+ `Content-Disposition`). The backup object carries `version`/`data` at the **top level** so restore validates directly (no outer envelope). **`Expense.receiptImage` (binary blob) is stripped from the export** (set to `undefined` right after the query) â€” a Bytes field serializes to a per-byte integer array (~4-6x size) and was the main cause of >50 MB backups; restore never reads it back. ## Production

Hetzner VPS (Hetzner cloud), Docker Compose. Stack defined in `docker-compose.prod.yml`:
- `budget-db-prod` (postgres:16-alpine, 512M), `budget-redis-prod` (redis:7-alpine, 320M â€” `maxmemory 256mb allkeys-lru`), `budget-api-prod` (1G, `NODE_OPTIONS=--max-old-space-size=768`), `budget-admin-prod` (256M), plus shared `accounting-nginx` (separate stack) reverse-proxying `api.ai-budget.pl` and `admin.ai-budget.pl`. API container has **no host port mapping** â€” reach it only through nginx or the docker network. **API memory (ABA-163)**: without an explicit `NODE_OPTIONS=--max-old-space-size`, Node auto-caps the V8 heap at ~half the container limit (~256M at 512M), which OOM-crashed the backup export. Compose sets heap to 768 and `deploy.resources.limits.memory` to 1G â€” keep them in sync (heap + ~150M non-heap must fit under the limit).
- Volumes: `ai-budget_postgres_data`, `ai-budget_redis_data` (preserve across deploys; never `docker volume prune` without filters).
- **DB backups (ABA-163, Phase 1)**: `.github/workflows/backup-db.yml` â€” nightly cron (`0 2 * * *` UTC, off-host so it survives a VPS outage) + `workflow_dispatch`. SSHes to the VPS, runs `pg_dump -Fc` inside `budget-db-prod`, streams it to the runner, validates it (size â‰¥10 KB + `pg_restore --list` object count), `age`-encrypts (asymmetric â€” runner holds only `AGE_PUBLIC_KEY`; the private key is **offline**, never in CI), and publishes it as a Release asset (`ai_budget-YYYY-MM-DD.dump.age`, tag `backup-YYYY-MM-DD`) in the private repo `BACKUP_REPO` via `BACKUP_REPO_TOKEN` (PAT, `contents:write`). Logic lives in `scripts/backup-db.sh` (dump+sanity+encrypt; mockable via `LOCAL_DUMP`) and `scripts/prune-backups.sh` (GFS retention 7 daily/4 weekly-Sunday/6 monthly-1st; mockable via `RELEASES_FILE`/`DRY_RUN`). Failure â†’ Telegram alert (same pattern as `uptime-check.yml`). Restore: `docs/ops/restore-runbook.md` (decrypt â†’ scratch-restore â†’ verify â†’ prod `pg_restore --clean --if-exists`). RPO â‰¤24 h. Redis is **not** backed up (cache/ephemeral). Receipt images are in-DB `Bytes`, so the dump captures them (and grows with them â€” revisit object storage if a dump exceeds a few hundred MB). Phases 2â€“4 (infra fault tolerance, HA, mobile local backup) deferred â€” see `docs/superpowers/specs/2026-05-29-postgres-backup-phase1-design.md`.
- Deploy: push to `development` triggers `.github/workflows/deploy.yml` â†’ SSH â†’ `scripts/deploy.sh` (`git reset --hard`, `npm install` with lock, build, `prisma migrate deploy`, `up -d --force-recreate api admin`, then `docker image prune -f` + `docker builder prune -af`). Verify-step polls `https://api.ai-budget.pl/api/v1/health` for up to 120 s. **Build-cache pruning matters**: images are built on the VPS each deploy and the build cache accumulated ~4 GB â†’ disk hit 89% (ABA-165); the per-deploy `builder prune` keeps it in check. `.github/workflows/infra-diagnostics.yml` (manual `workflow_dispatch`) dumps a `docker system df` + `du` breakdown over SSH when disk needs investigating.
- **Snap-installed Docker is held and disabled** (`snap refresh --hold docker`, `snap disable docker`) after the 2026-04-27 incident where snap auto-refresh hijacked `/var/run/docker.sock` from the apt-installed daemon. Do not re-enable snap docker; only the apt-installed `dockerd` (system data root `/var/lib/docker`) owns prod state.
- Adding env vars: `docker restart` does NOT reload `env_file`. Must `docker compose -f docker-compose.prod.yml --env-file .env.production up -d --force-recreate <service>`.
- **CORS (ABA-151)**: `main.ts` builds `corsOrigins` from `process.env.CORS_ORIGIN.split(',')` (falls back to `localhost:8081`/`:3001` when unset). It must be an **explicit comma-separated origin list** in prod `.env.production` â€” e.g. `CORS_ORIGIN=https://admin.ai-budget.pl,https://ai-budget.pl`. Do **NOT** use `CORS_ORIGIN=*`: it becomes the array `["*"]`, which Express/Nest `cors` matches as a literal origin (not a wildcard), so the admin browser gets no `Access-Control-Allow-Origin` and login silently breaks (and `*` is invalid with `credentials:true`). Native mobile is unaffected (no `Origin` header). After changing it, force-recreate the `api` service (see env-var note above).

## Mobile releases (Android)

A silent backup failure is the classic way to discover at restore time that there are no backups â€” so failures, not successes, are alerted.
- **Infra watch (ABA-165)**: `.github/workflows/infra-watch.yml` â€” cron every 30 min, SSHes to the VPS and pipes `scripts/infra-check.sh` (`ssh â€¦ 'bash -s' < script`) to check root-disk usage (> 85% alerts) and that all 4 prod containers are running and not `unhealthy`; non-zero exit â†’ Telegram alert. - **Log rotation (ABA-165)**: `docker-compose.prod.yml` sets `logging` `json-file` `max-size: 10m` / `max-file: 3` on postgres/redis/api/admin (~30 MB/container) so logs can't fill the disk.
