# AI Budget Assistant

## Architecture

Turborepo monorepo with 5 packages:

| Package | Tech | Purpose |
|---|---|---|
| `apps/api` | NestJS 10 + Prisma 5 + PostgreSQL + Redis | REST API backend |
| `apps/mobile` | Expo 54 + React Native 0.81 + Zustand + SQLite/Drizzle | Mobile app (iOS/Android/Web) |
| `apps/admin` | Next.js 16 + React 19 + TailwindCSS 4 + shadcn/ui + Recharts | Admin web dashboard (port 3001) |
| `packages/shared-types` | TypeScript interfaces | Entities and DTOs shared between api and mobile |
| `packages/shared-utils` | Zod schemas + formatting + constants | Validation and utilities shared between api and mobile |

## Key Patterns

### API (NestJS)
- **Module structure**: `modules/<feature>/` contains `module.ts`, `controller.ts`, `service.ts`, `dto/index.ts`, `guards/`
- **Auth**: JWT via `@UseGuards(JwtAuthGuard)`. Request type: `AuthenticatedRequest` from `common/types/index.ts`. Password reset via 6-digit email code (`POST /auth/forgot-password`, `POST /auth/reset-password`) with in-memory rate limiting and bcrypt-hashed codes. Email change also via 6-digit code (`POST /auth/change-email/request`, `POST /auth/change-email/confirm`) — JWT-guarded, requires current password on request, issues new tokens on confirm
- **Account scoping**: `AccountContextGuard` middleware reads `X-Account-Id` header, resolves membership, adds `accountId` and `accountRole` to request
- **Role-based access**: `AccountRoleGuard` with `@RequireRole('owner'|'editor')` decorator (DI-based, needs `AccountsModule`). `ViewerBlockGuard` (ABA-155) — zero-dependency guard in same file; use as `@UseGuards(new ViewerBlockGuard())` on any write method without importing `AccountsModule`. Reads `req.accountRole` set by `AccountContextGuard`. Apply to all POST/PATCH/PUT/DELETE endpoints that mutate account-scoped data. AI chat: viewer write-actions blocked in `chat.service.ts` before a pending action is queued. Bots: `TelegramUserState` and `WhatsAppUserState` include `accountRole`; link services join `AccountMember` on `getLink()`; write handlers check the role before executing.
- **Service signature**: `(accountId, userId, dto)` as parameters, all Prisma queries filter by `accountId`
- **Transaction attribution**: `Expense.userId` and `Income.userId` already store the creator's id (set from `request.userId` at create time). Both services include `user: { select: { name: true } }` in all Prisma queries and call a `toExpenseResponse()`/`toIncomeResponse()` helper to flatten it into `createdByUserName: string | null` on the returned object. No migration needed — just always pass the user include and map the result.
- **Database**: Prisma ORM. Schema at `apps/api/prisma/schema.prisma`. Uses `@map("snake_case")` for column names. Prod `DATABASE_URL` includes `connection_limit=10` (see `docker-compose.prod.yml`) to cap the connection pool.
- **32 modules**: `accounts`, `account-transfers`, `admin`, `ai`, `analytics`, `app-versions`, `auth`, `backups`, `budgets`, `categories`, `currency-exchange`, `debts`, `encryption`, `expenses`, `gamification`, `import-bank`, `import-batches`, `import-wise`, `incomes`, `insights`, `investments`, `mail`, `notifications`, `projects`, `referrals`, `reports`, `subscriptions`, `sync`, `tags`, `telegram`, `users`, `wallet`
- **Caching layer** (ABA-151): `common/cache/cache.service.ts` — global `@Global()` ioredis wrapper; `delByPrefix` uses cursor-based `SCAN` (not blocking `KEYS`). `common/cache/redis-throttler-storage.ts` — `RedisThrottlerStorage` (`ThrottlerStorage` v5 interface, INCR+PEXPIRE NX+PTTL pipeline, `keyPrefix: 'throttle:'`); registered via `ThrottlerModule.forRootAsync` in `app.module.ts` so rate limits survive restarts.
- **Debt reminders**: `modules/debts/debt-reminder.cron.ts` — `@Cron('0 9 * * *')` daily cron that finds active lent/borrowed debts due in 3 days (upcoming) or overdue yesterday, batches repayment queries to compute remaining balance, and calls `NotificationsService.sendToUser(..., 'debt_reminder')`. Respects `user.notifyDebtReminders` preference (default `true`). User preference toggle in `app/settings/notifications.tsx`. `NotificationType` includes `'debt_reminder'`; `NotificationPreferencesResponse` includes `debtReminders: boolean`.
- **Recurring expenses**: `modules/expenses/expense-recurring.cron.ts` — `@Cron('0 8 * * *')` daily cron that groups all active recurring expenses by `recurringId`, takes the most recent per series, computes `nextDue = lastDate + period` (weekly/monthly/yearly), and if `nextDue <= today` clones the expense with today's date (same `recurringId`) and sends a push notification. Respects `user.notifyRecurringExpenses` preference (default `true`). Mobile new-expense form has a "Repeat" toggle + period chip picker that sets `isRecurring: true`, assigns a `recurringId` UUID, and sets `recurringPeriod`. Expense detail screen shows a "Part of a recurring series" banner with period label and an inline "Stop Recurring" button (`PATCH /expenses/:id/stop-recurring` sets `isRecurring: false`; history preserved). User preference toggle in `app/settings/notifications.tsx`.
- **App version gate**: `modules/app-versions/` — public `GET /app-versions/check?platform=ios|android&version=x.y.z` (no auth, mobile calls before login) returns `{latestVersion, minSupportedVersion, isUpdateAvailable, isUpdateRequired, releaseNotes, storeUrl}`. Admin CRUD under `/admin/app-versions` (behind `JwtAuthGuard + AdminGuard`). Latest row per platform = `ORDER BY publishedAt DESC LIMIT 1`. Service enforces `latestVersion >= minSupportedVersion` via `utils/semver.ts`. Default Play Store URL hardcoded; iOS storeUrl is a placeholder until App Store ID is assigned (admin overrides per row).
- **Budget history**: `GET /budgets/:id/history?periods=6` returns chronological `BudgetHistoryEntry[]` ({periodStart, periodEnd, limit, actual, isOverBudget}). Uses `computeBudgetPeriod` to iterate the last N periods (max 12). Returns `[]` for `custom`-period budgets. Category / currency filters mirror `getProgress`. Mobile stores result in `budgetStore.budgetHistory[budgetId]` (in-memory, fetched on budget detail screen mount).
- **Budget category model**: `Budget` has **no** `categoryId` field. Category filtering is done exclusively through `BudgetCategory` allocations (`budget_categories` table). An overall budget (no category filter) has zero `BudgetCategory` rows. A single-category budget has exactly one row with `amount = budget.amount`. Multi-category budgets have multiple rows. The legacy `category_id` column was removed via migration `20260523000000_drop_budget_category_id` (backfill + drop). Mobile `budgets` SQLite table still has a dead `category_id` column on existing devices but it is never read or written.
- **Wise CSV import**: `modules/import-wise/` — `POST /import/wise/preview` (multipart CSV ≤5 MB, `papaparse`, strips BOM) classifies rows as `expense` (Amount<0) / `income` (Amount>0) / `fx` (paired conversion rows sharing `Payment Reference + Date + opposite sign`, emitted as `CurrencyExchange` not `AccountTransfer` since Wise FX is in-wallet), folds `Total fees` into amount, suggests categories via `MERCHANT_CATEGORY_HINTS`. `POST /import/wise/commit` writes in one Prisma `$transaction` with `source: 'import'`, `externalRef = 'wise:<TransferWise ID>'`. **Dedup**: `Expense`/`Income`/`CurrencyExchange` each have `externalRef String?` with `@@unique([accountId, externalRef])` (NULLs distinct in Postgres). Mobile `app/settings/wise-import.tsx` (5-state machine, `expo-document-picker`); reached via the `/settings/import` hub.
- **Polish bank import** (ABA-126): `modules/import-bank/` — strategy-registry module for CSV/PDF statement imports. Banks: `mbank`, `pko`, `ing`, `millennium`, `pekao` + `universal` column-mapping fallback. 5 endpoints under `/import/bank/*` (`JwtAuthGuard + AccountContextGuard`): `POST preview` (multipart ≤5 MB), `POST commit`, `GET/POST mappings`, `DELETE mappings/:id`. `ImportBankService` flow: `decodeCsvBuffer` (UTF-8/Windows-1250 auto-detect, `iconv-lite`) → parser dispatch (mappingId → bankId → saved-fingerprint → auto-detect) → normalized rows → `pairFxRows` (same date, opposite sign, different currency) → `buildExternalRef` → dedup. Per-bank parsers `parsers/*.parser.ts` implement `BankParser { id, displayName, detect(), parse() }`; `universal.detect()` always returns `false`. Helpers in `utils/`: `polish-amount.ts`, `polish-date.ts`, `encoding.ts`, `header-fingerprint.ts`, `fx-pairing.ts`, `pdf-text.ts`. `merchants/merchants-pl.ts` ~40 brand→category hints. **Dedup key**: `bank:<bankId>:<isoDate>:<signedAmountCents>:<sha256(normalize(desc)).slice(0,8)>`. **Two dedup layers** in `buildPreviewResponse`: (1) exact `externalRef` match (re-import of same file); (2) `flagContentDuplicates` — content match on `(date, signedAmountCents, currency)` against ALL account Expense/Income regardless of source, greedy 1-to-1 multiset, excludes FX. Matched rows → `alreadyImported: true`, auto-unchecked in preview. `Expense.date`/`Income.date` are `@db.Date` so date match is exact. **Saved mappings**: `csv_import_mappings` table (migration `20260523100000_add_csv_import_mappings`, `@@unique([accountId, headerFingerprint])`); commit `saveMapping` field persists column mapping for auto-application. **PDF statements**: `BankParser.format?: 'csv'|'pdf'`; orchestrator detects `%PDF` header (`isPdfBuffer`), extracts text via `pdf-parse` (`extractPdfText`), dispatches to PDF parsers — CSV header/mapping/fingerprint skipped. `erste.parser.ts` and `alior.parser.ts` are PDF parsers (line state machines over the bank's "Wyciąg" table). **Request-a-bank**: `POST /import/bank/request-bank` (multipart `file?` + `bankName`, `notes?`) → forwards to **ops Telegram chat** (`TELEGRAM_CHAT_ID`, never the user) via `TelegramService.sendMessage`/`sendDocument`. Mobile: Settings → "Import transactions" → `app/settings/import/` (`index.tsx` hub, `preview.tsx`, `mapper.tsx`, `request-bank.tsx`); shared `useImportStore`; `bankImport.*` i18n (8 locales); help section `27-bank-import`. **Visible banks**: Wise, mBank, PKO, Erste (PDF), Alior (PDF), Other — ING/Millennium/Pekao are in the registry but hidden in `BANKS` (`app/settings/import/index.tsx`) until validated against real exports. mBank/PKO parsers were rewritten against real exports (mBank: metadata preamble before `#Data operacji…#Kwota`, amounts without ` PLN`; PKO: comma-delimited quoted, single signed `Kwota`, array-mode parse).
- **Import batch history** (ABA-130): `modules/import-batches/` — `GET /import/batches` (last 20, each with `canRollback`: `status === 'committed'` and within 30-day window); `DELETE /import/batches/:id` rolls back (sets `isDeleted=true` + `externalRef=null` on linked rows so the file can be re-imported, marks batch `rolled_back`). Both wise + bank commit endpoints create an `ImportBatch` in-transaction and stamp records with `importBatchId`. Table `import_batches` (migration `20260525144711_add_import_batches`); nullable `import_batch_id` FK on `expenses`/`incomes`/`currency_exchanges`. Mobile: "Past imports" section in `app/settings/import/index.tsx`. `ImportBatchesModule` imported by both `ImportWiseModule` and `ImportBankModule`.
- **Backups** (ABA-163): `modules/backups/` — `POST /backups/export` builds a full account snapshot, serializes it **once**, and streams the JSON string straight to the response body via `@Res()` (no Nest re-serialization — returning `data: backup` previously tripled peak memory and OOM-crashed the API). Filename travels in the `X-Backup-Filename` header (+ `Content-Disposition`). The backup object carries `version`/`data` at the **top level** so restore validates directly (no outer envelope). Mobile calls the endpoint **once** (`reports.api.ts` `downloadBackupData()` → `{ blob, fileName }`); `reportStore.exportBackup()` returns a `BackupExportResult` status so the UI only claims success when the file is actually saved/shared (Android SAF folder picker → shows path; else system share sheet). `POST /backups/restore` upserts by `clientId`/name, `GET /backups/history`. 50 MB cap. See also API-memory note under Production.
- **AI module features**:
  - **Chat Q&A**: Natural language financial questions powered by GPT-4
  - **Natural Language Commands**: Execute actions via chat (create expenses/budgets, query data) using OpenAI function calling
  - **11 AI functions**: `create_expense`, `create_income`, `create_budget`, `create_category`, `get_expenses`, `get_budget_status`, `get_category_breakdown`, `record_debt_repayment`, `create_debt`, `get_debt_summary`, `update_goal_balance`
  - **Debt commands**: `record_debt_repayment(debtId, amount, date?)` creates a linked repayment income/expense; `create_debt(contactName, amount, currencyCode, direction, dueDate?)` creates lent (Expense) or borrowed (Income) record with `isDebt=true`; `get_debt_summary()` returns active debts (read, cached). AI resolves contact names via `activeDebts` context (ids included) — ambiguous names trigger clarifying question.
  - **Goal command**: `update_goal_balance(goalId, newAmount)` calls `GoalPlannerService.updateGoal({ currentAmount })`. Resolves goal names via `savingsGoals` context (ids now included). Auto-completes goal if `currentAmount >= targetAmount`.
  - **Confirmation flow**: Write actions (`create_*`, `record_debt_repayment`, `update_goal_balance`) require user confirmation before execution; read actions (`get_*`) execute immediately (10-min cache)
  - **Language detection**: Automatically detects user language (Russian, Ukrainian, Belarusian, German, Spanish, French, Polish, English) and responds in same language
  - **Currency mapping**: Supports currency symbol detection (₴→UAH, $→USD, €→EUR, zł→PLN, £→GBP, ₽→RUB)
  - **Endpoints** (all behind `JwtAuthGuard + AccountContextGuard` via the class-level decorator on `AiController`): `POST /ai/chat`, `POST /ai/chat/confirm`, `POST /ai/chat/reject`; `GET /ai/chat/conversations` (last 20, **account-scoped**: `accountId = X-Account-Id AND (isShared OR userId = me)`), `GET /ai/chat/conversations/:id/messages` (last 50, user+assistant roles only, same access predicate), `GET /ai/chat/conversations/:id/poll?since=<iso>` (newer messages + refreshes Redis presence), `PATCH /ai/chat/conversations/:id/shared` (owner-only sharing toggle). `confirm`/`reject` verify the conversation is in the caller's account AND that the caller initiated the `pending_action` (scoped by its `senderUserId`).
  - **Chat history**: `ChatConversation` (now `accountId` + `isShared`) + `ChatMessage` (now `senderUserId` + `mentionedUserIds`) stored server-side. Mobile loads history via `chatStore.loadConversations()` / `loadConversation(id)` with SQLite cache (`src/db/chatRepository.ts`, account-scoped local query). Chat tab shows a History bottom-sheet and a "New Conversation" button when browsing past conversations.
  - **Shared conversations**: per-conversation opt-in group chat for shared accounts. `ChatConversation.isShared` (default false; **owner-only** to set, via PATCH endpoint or `chat()`'s `initialIsShared` gated on `accountRole === 'owner'`) makes it visible to all members; private stays creator-only. A message that `@mentions` a member (mentions passed as `{userId}[]`, validated, self excluded) **silences the AI** and pushes a `chat_mention` notification (gated by `notifySharedActivity`) to each mentioned member **not currently present**; no mention → AI replies. Presence in Redis via `CacheService` key `chat:presence:{conversationId}:{userId}` (TTL 45s). AI history prefixes each member's message with sanitized `[Name]: `. Mobile polls `…/poll?since=` every 4s (only while a shared conversation is focused), reconciles optimistic id to server `userMessageId`, renders others' messages left-aligned with name label, `@`-mention suggestion bar, owner-only Shared/Private header toggle. **Deep-link**: tapping a `chat_mention` push → `handleNotificationResponse` switches `accountId` → `router.push('/(tabs)/chat', { conversationId })`; chat screen reads the param and `loadConversation` (falls back to `loadConversations()` when uncached).
  - **AI service split** (ABA-119): `chat.service.ts` is the lean orchestrator (415 lines); three focused helpers live alongside it: `user-context-builder.service.ts` (builds `UserContext` from Prisma/domain services), `ai-tools.service.ts` (all 11 OpenAI function schemas + `executeAction` dispatcher + cache wrapper), `prompt-builder.service.ts` (system prompt construction, language detection, action i18n). All three are registered in `AiModule` as providers but not exported.
  - **UserContext caching** (ABA-151): `UserContextBuilder.build()` caches the result in Redis under key `uc:{accountId}` (TTL 60 s). `ExpensesService.invalidateChatCache()` and `IncomesService` mutation methods (`create`/`update`/`remove`) call `CacheService.del('uc:{accountId}')` after writing so the cache is refreshed promptly. The exported `ucKey(accountId)` helper in `user-context-builder.service.ts` is the canonical key formula.

- **Telegram bot**: `modules/telegram/` — Telegraf-based bot with 7 handlers: `ChatHandler` (AI chat with usage tracking, 1.0/msg), `VoiceHandler` (Whisper transcription + chat, 2.0), `PhotoHandler` (OCR receipt scan, 2.0), `CommandHandler` (/start, /link, /help, /usage, /account, /newchat, /unlink), `ExpenseHandler`, `IncomeHandler`, `CategoryHandler`. All system messages localized via `helpers/i18n.ts` (8 languages, resolved from `user.language`). AI usage tracked and limits enforced — users get localized warning when limit reached.

- **WhatsApp bot**: `modules/whatsapp/` — `@Global()` module on the Meta Business Cloud API. Parallel to Telegram, same features + shared services (`ChatService`, `WhisperService`, `OcrService`, `ExpensesService`, `IncomesService`, `CategoriesService`, `SubscriptionsService`). Differences: (a) **webhook-only** `POST /whatsapp/webhook`, excluded from `/api/v1` prefix in `main.ts`; (b) **HMAC-SHA256** verification via `verifySignature` over `req.rawBody`; (c) **Redis state** not in-memory — `wa:msg:{id}` (idempotency 24h), `wa:pa:{shortId}` (pending actions 1800s), `wa:receipt:{shortId}` + `wa:awaiting_date:{phone}`, `wa:cat:{shortId}`; (d) callback IDs use `--` separator (UUIDs contain single `-`); (e) interactive via `WhatsAppClientService.sendButtons` (max 3×20char) / `sendList` (max 10 rows); (f) WA markdown (`*bold*`, `_italic_`) via `markdownToWhatsApp`. Linking via 6-hex code: mobile QR + `wa.me/{phone}?text=link%20{code}` in the **Chat bots** screen `app/settings/bots.tsx`; `CommandHandler.handleLink` is the only command from unlinked numbers. Endpoints on `UsersController` (`POST/GET/DELETE /users/me/whatsapp-link[-code]`). `helpers/i18n.ts` ports telegram's (33 keys × 8 languages).

### Mobile (React Native/Expo)
- **Platforms**: iOS, Android, **Web** (Expo web via `react-native-web` + Metro). Web is supported for quick browser testing — run `npm run dev:web` from project root (or `npm run web` from `apps/mobile/`). On web, native-only modules fall back via platform-specific files: `secureStorage.web.ts` (localStorage), `db/client.web.ts` (in-memory mock, no persistence — server still works), `useBiometric.web.ts` (no-op). MMKV uses its built-in `createMMKV.web.ts` (localStorage-backed). `expo-notifications`, `react-native-android-widget`, `expo-screen-orientation` are guarded by `Platform.OS === 'web'` checks or platform-specific imports. **Caveat**: SQLite-backed offline-first flows are disabled on web — data shows only what the API returns, no local cache; receipts/voice/biometric features are also degraded. Use web for UI testing, not full functional testing.
- **Navigation**: Expo Router. Screens in `app/`, tabs in `app/(tabs)/` — home, expenses, budgets, analytics, chat. `(tabs)/_layout.tsx` fires `Haptics.selectionAsync()` on every `tabPress` via `screenListeners` (`expo-haptics`, web no-op).
- **Keyboard handling**: **pure-JS, no native module** (builds with the local Windows Android toolchain — `react-native-keyboard-controller`'s Fabric codegen blows past the Windows 260-char MAX_PATH limit; do NOT reintroduce it or any keyboard native module). `src/hooks/useKeyboardHeight.ts` tracks height via RN `Keyboard` events (works under New Arch + Android edge-to-edge, where `KeyboardAvoidingView`/`adjustResize` are unreliable). Two wrappers in `src/components/`: `KeyboardAwareScreen` (drop-in for a form's root `ScrollView`: Android pads content by keyboard height, iOS `automaticallyAdjustKeyboardInsets`) and `KeyboardAvoidingScreen` (drop-in for `KeyboardAvoidingView` — pads bottom to lift docked inputs/footers; accepts+ignores `behavior`). Form screens import `KeyboardAvoidingScreen as KeyboardAvoidingView`.
- **State**: 23 Zustand stores in `src/stores/` — `authStore`, `accountStore`, `expenseStore`, `incomeStore`, `budgetStore`, `categoryStore`, `tagStore`, `projectStore`, `walletStore`, `chatStore`, `insightsStore`, `exchangeRateStore`, `subscriptionStore`, `themeStore`, `widgetVisibilityStore`, `debtStore`, `encryptionStore`, `gamificationStore`, `goalStore`, `investmentStore`, `referralStore`, `reportStore`, `scenarioStore`
- **Local DB**: SQLite via Drizzle ORM. Schema in `src/db/schema/index.ts`. 18 repositories in `src/db/*Repository.ts` use raw `executeSql()` — `account`, `accountTransfer`, `budget`, `budgetCategory`, `category`, `chat`, `currencyExchange`, `encryption`, `expense`, `expenseItem`, `gamification`, `income`, `investment`, `project`, `split`, `syncMetadata`, `tag`, `wallet`
- **API client**: `src/services/api.ts` — barrel that re-exports a composed `api` singleton. Base infrastructure lives in `src/services/http-client.ts` (`HttpClient` class: `request()`, `getAuthToken()`, `refreshToken()`, `setAccountIdGetter()`, `setLogoutHandler()`). Domain methods are split across 14 files: `auth.api.ts`, `users.api.ts`, `expenses.api.ts`, `incomes.api.ts`, `budgets.api.ts`, `categories.api.ts`, `analytics.api.ts`, `ai.api.ts`, `accounts.api.ts`, `wallet.api.ts`, `investments.api.ts`, `encryption.api.ts`, `reports.api.ts`, `subscriptions.api.ts`. All stores import `{ api }` from `api.ts` unchanged. Auto-injects `X-Account-Id` header, auto JWT refresh, 401 → logout.
- **Offline-first**: write to SQLite first, queue sync via `syncQueue` table, sync to server when online. `SyncService.pushChanges()` processes the `changes[]` array in parallel batches of 10 (ABA-152) so large mobile resyncs are faster without contention from unbounded parallelism. **Logging (ABA-157)**: a failed server **push** while offline is an *expected* outcome (the row stays `pending` and re-syncs later) — log it with `console.warn`, NEVER `console.error`. RN's dev LogBox renders any `console.error` as a blocking full-screen red overlay, which users perceive as an offline-add crash. The `.catch` on fire-and-forget `api.create*/update*/delete*` calls in `expenseStore`/`incomeStore` uses `warn`; only genuine local-SQLite failures stay `console.error`. (`no-console` ESLint rule allows `warn`/`error`.)
- **Expense ↔ project association**: an expense's `projectId` is **not** an `expenses` column — it lives in the `project_expenses` join table (mirrored on server as `ProjectExpense`). The generic `updateExpense(id, updates)` store action canNOT manage it (the field isn't a column, and `undefined` is dropped from the JSON body so clearing is impossible). Use `expenseStore.setExpenseProject(expenseId, projectId | null)` instead — it updates in-memory state, the local SQLite join table (`addExpenseToProject`/`removeExpenseFromProject` in `projectRepository`), and the server via `api.updateExpense(id, { projectId })` (explicit `null` clears; `UpdateExpenseDto.projectId` accepts `string | null`, service soft-deletes the old row). Both `expense/new.tsx` (create) and `expense/[id].tsx` (edit) render the shared `ProjectPicker` component.
- **Expense merchant field** (ABA-140): `Expense.merchant?: string` is a plain free-text column (Prisma `merchant String?` + `@@index([accountId, merchant])`; mobile SQLite `merchant TEXT`). Auto-populated from receipt OCR (mobile `expense/receipt.tsx`, Telegram/WhatsApp `photo.handler.ts`) and bank/Wise import commit; manually editable via the shared `MerchantInput` component (free text + autocomplete suggestions from `expenseStore.getDistinctMerchants()`). Encrypted client-side **like `description`** — it's in `ENCRYPTION_FIELDS.expense.tier1`, so push paths (`addExpense`, `syncPendingExpenses`) run it through `maybeEncrypt` and the `loadExpenses` pull merge reads `decrypted.merchant`. Shown on the detail screen + as a secondary line on Expenses-tab rows. Filtering is **client-side only** (no `?merchant=` API param): `ExpenseFilters.merchants: string[]` (multi-select, default `[]`) — matches any of the selected merchants; the search box also matches merchant substring. The Expenses-tab category + merchant filters share **one filter row** (ABA-142): `[Category ▾] [Merchants N ▾]  ……  −total`. The merchant picker is **multi-select** (toggle stays open; "All merchants" clears; "Done" closes). The right-aligned total is `sumConverted(filteredItems, baseCurrency, rates)` from `src/utils/total.ts`. Income tab shows `[Category ▾] …… +total` (no merchant pill). Clearing on edit sends `''` (treated as absent everywhere). No backfill; income has no merchant field.
- **Merchant management + capture reconciliation** (ABA-141): Settings → **Merchants** screen (`app/settings/merchants.tsx`, header registered in `app/_layout.tsx`) lists distinct merchants with counts (`expenseStore.getMerchantCounts()` → `getMerchantCounts` util) and supports **rename / merge / delete**: `expenseStore.renameMerchant(from, to | null)` (rename-to-existing merges; `null` clears) updates matching in-memory rows + one `bulkRenameMerchant` SQL `UPDATE` (account-scoped, marks rows `pending`) then fires `syncPendingExpenses()` (re-encrypts for E2EE). **Capture reconciliation**: receipt OCR (`expense/receipt.tsx`) and voice (`expense/voice.tsx`) pre-fill the merchant via `resolveExistingMerchant(input, getDistinctMerchants())` (exact case-insensitive/trim match snaps to the existing canonical value) and render the editable `MerchantInput` (suggest+confirm; no fuzzy auto-map). **Voice now stores its merchant in `merchant`, not `notes`** (legacy voice rows keep it in `notes`, not retro-migrated). All three reconciliation/management helpers live in `src/utils/merchant.ts`.
- **Local-first tab hydration**: list-bearing tabs (`(tabs)/index`, `expenses`, `analytics`) call `hydrateTransactions()` (in `src/stores/hydrateTransactions.ts`) from `useEffect([currentAccountId])`. It runs `loadExpenses` then `loadIncomes` **sequentially** (parallel contends on the single SQLite connection — reads went 65ms→65-602ms spiky) and exposes `useHydrationStore.isHydrating`. Its re-entry guard collapses the ~5 parallel call sites to one cycle. Pull-to-refresh and `settings/data.tsx` "Sync now" pass `{ force: true }`. `loadExpenses`/`loadIncomes` each also have a re-entry guard + 30s per-account skip window (`force` bypasses). Server-pull merge is 4 phases: A) dedup unique categories/projects, B) decrypt rows in parallel (`Promise.all`), C) build entities in JS, D) one `withTransaction` for ALL writes. ## Dependency Order for Changes

When modifying features that span multiple packages, follow this order:

1. `packages/shared-types` — entity interfaces and DTOs
2. `packages/shared-utils` — Zod validation schemas (if needed)
3. `apps/api/prisma/schema.prisma` — database schema, then `npx prisma migrate dev --name <name>` + `npx prisma generate`
4. `apps/api/src/modules/*` — API services, controllers, guards

## Production

Hetzner VPS (Hetzner cloud), Docker Compose. Stack defined in `docker-compose.prod.yml`:
- `budget-db-prod` (postgres:16-alpine, 512M), `budget-redis-prod` (redis:7-alpine, 320M — `maxmemory 256mb allkeys-lru`), `budget-api-prod` (1G, `NODE_OPTIONS=--max-old-space-size=768`), `budget-admin-prod` (256M), plus shared `accounting-nginx` (separate stack) reverse-proxying `api.ai-budget.pl` and `admin.ai-budget.pl`. API container has **no host port mapping** — reach it only through nginx or the docker network. **API memory (ABA-163)**: without an explicit `NODE_OPTIONS=--max-old-space-size`, Node auto-caps the V8 heap at ~half the container limit (~256M at 512M), which OOM-crashed the backup export. Compose sets heap to 768 and `deploy.resources.limits.memory` to 1G — keep them in sync (heap + ~150M non-heap must fit under the limit).
- Volumes: `ai-budget_postgres_data`, `ai-budget_redis_data` (preserve across deploys; never `docker volume prune` without filters).
- Deploy: push to `development` triggers `.github/workflows/deploy.yml` → SSH → `scripts/deploy.sh` (`git reset --hard`, `npm install` with lock, build, `prisma migrate deploy`, `up -d --force-recreate api admin`). Verify-step polls `https://api.ai-budget.pl/api/v1/health` for up to 120 s.
- **Snap-installed Docker is held and disabled** (`snap refresh --hold docker`, `snap disable docker`) after the 2026-04-27 incident where snap auto-refresh hijacked `/var/run/docker.sock` from the apt-installed daemon. Do not re-enable snap docker; only the apt-installed `dockerd` (system data root `/var/lib/docker`) owns prod state.
- Adding env vars: `docker restart` does NOT reload `env_file`. Must `docker compose -f docker-compose.prod.yml --env-file .env.production up -d --force-recreate <service>`.
- **CORS (ABA-151)**: `main.ts` builds `corsOrigins` from `process.env.CORS_ORIGIN.split(',')` (falls back to `localhost:8081`/`:3001` when unset). It must be an **explicit comma-separated origin list** in prod `.env.production` — e.g. `CORS_ORIGIN=https://admin.ai-budget.pl,https://ai-budget.pl`. Do **NOT** use `CORS_ORIGIN=*`: it becomes the array `["*"]`, which Express/Nest `cors` matches as a literal origin (not a wildcard), so the admin browser gets no `Access-Control-Allow-Origin` and login silently breaks (and `*` is invalid with `credentials:true`). Native mobile is unaffected (no `Origin` header). After changing it, force-recreate the `api` service (see env-var note above).

## Mobile releases (Android)

- **Bare Expo workflow** — `apps/mobile/android/` is committed, so EAS builds from the native project and does **not** run prebuild or read `app.json` for the version. The version shown in **Google Play comes from `apps/mobile/android/app/build.gradle` `versionName`**, not `app.json`. `eas.json` `appVersionSource: "remote"` + production `autoIncrement: true` only manage the integer `versionCode` (build number) remotely; they never touch `versionName`. (Historically this caused every release to ship as `1.0.0` while `versionCode` quietly incremented — ABA-131.)
- **Before each release**: bump `versionName` in `android/app/build.gradle` (source of truth for Play) AND `version` in `app.json` (kept in sync for the in-app update gate) to the same value. Do not hand-edit `versionCode` — EAS remote autoIncrement owns it.
- **CI guard**: `apps/mobile/scripts/check-release-version.js` runs as a production-only step in `mobile-build.yml` and `mobile-eas-build.yml`. It fails the build if `build.gradle` versionName ≠ `app.json` version, and (best effort) if the version isn't strictly greater than the latest `v*.*.*` git tag (skipped when no release tags exist).
- **Publish**: run the *Mobile Build & Publish* workflow (`mobile-build.yml`, profile `production`, submit `true`) → builds AAB `--local` → `eas submit` to the Play production track. After release, also add the new version in **admin → App Versions** so the in-app `UpdatePrompt` knows about it.

## Observability

- **Health**: `GET /api/v1/health` (public, no auth) — runs `SELECT 1` on Postgres, returns `{status, db, uptimeSeconds, timestamp}`. 503 if DB fails. Used by Docker `HEALTHCHECK` (requires HTTP 200) and CI verify-step.
- **Uptime**: `.github/workflows/uptime-check.yml` runs every 5 min via cron, hits the public `/api/v1/health`. On non-200, sends a Telegram alert via `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` GitHub Secrets with HTTP code, response body, and a link to the failed run.
- **Errors**: `@sentry/node` v8 captures unhandled exceptions and Express 5xx errors. Init must remain at the top of `main.ts` via `import './instrument'` — moving it after other imports breaks instrumentation.
- **No Prometheus / Datadog / external APM** at this time.
