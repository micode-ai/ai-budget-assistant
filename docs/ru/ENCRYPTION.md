# Сквозное шифрование (E2EE)

## Обзор

Сквозное шифрование гарантирует, что конфиденциальные финансовые данные (описания, заметки, названия) шифруются на устройстве пользователя до отправки на сервер. Сервер хранит только зашифрованные блобы и не может прочитать содержимое. Это защищает данные даже при компрометации серверной базы данных.

## Руководство пользователя

### Настройка шифрования

1. Откройте **Настройки** в приложении
2. Прокрутите до раздела **Безопасность**
3. Нажмите **Включить шифрование**
4. Введите **парольную фразу шифрования** (минимум 8 символов)
   - Это отдельный пароль от пароля для входа
   - Сервер никогда не видит эту фразу
5. Подтвердите парольную фразу
6. **Сохраните ключ восстановления**, который появится на экране
   - Формат: `XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX-XXXX`
   - Сохраните его офлайн (запишите, сохраните в менеджере паролей)
   - Это единственный способ восстановить данные при потере парольной фразы

### Разблокировка шифрования

После перезапуска приложения или истечения сессии шифрование блокируется. Для разблокировки:

1. Откройте **Настройки** > **Безопасность**
2. Нажмите **Разблокировать шифрование**
3. Введите парольную фразу
4. Зашифрованные данные снова станут читаемыми

### Что шифруется

| Уровень | Зашифрованные поля | Серверные функции |
|---------|-------------------|-------------------|
| **Уровень 1** (по умолчанию) | описания, заметки, названия, локации, чеки | Аналитика, бюджеты, AI-инсайты работают нормально |
| **Уровень 2** (опционально) | Всё из уровня 1 + суммы, цены, курсы обмена | Серверная аналитика недоступна; вычисляется локально |

### Восстановление

Если вы забыли парольную фразу:

1. Откройте **Настройки** > **Безопасность** > **Восстановить**
2. Введите ключ восстановления
3. Установите новую парольную фразу
4. Будет сгенерирован новый ключ восстановления (сохраните его снова)

### Сброс шифрования

Если потеряны и парольная фраза, и ключ восстановления:

1. Откройте **Настройки** > **Безопасность**
2. Нажмите **Сбросить шифрование** (красная кнопка)
3. Подтвердите действие
4. **Внимание**: Ранее зашифрованные данные на сервере станут навсегда нечитаемыми
5. Можно настроить шифрование заново с новой парольной фразой

### Общие аккаунты

Когда для общего аккаунта включено шифрование:

- Владелец аккаунта должен **выдать ключи шифрования** каждому участнику
- Новые участники видят метаданные (суммы, даты, категории), но не могут прочитать зашифрованные текстовые поля, пока владелец не выдаст доступ
- Выдача ключа происходит автоматически, когда владелец открывает приложение и подтверждает ожидающих участников

## Техническая документация

### Криптографические примитивы

| Назначение | Алгоритм | Библиотека | Платформа |
|-----------|----------|------------|-----------|
| Симметричное шифрование | XSalsa20-Poly1305 (NaCl secretbox) | tweetnacl | Все |
| Деривация ключей | PBKDF2-HMAC-SHA256, 100K итераций | Чистый JS (свой) | Все |
| Хеширование SHA-256 | SHA-256 | Чистый JS (свой) | Все |
| Обмен ключами (ECDH) | X25519 | tweetnacl | Все |
| Цифровые подписи | Ed25519 | tweetnacl | Все |
| Случайные байты | `ExpoCrypto.getRandomBytes` / `crypto.getRandomValues` | expo-crypto / Web Crypto | Native / Web |

**Почему чистый JS SHA-256?** PBKDF2 со 100K итерациями требует ~200K вызовов SHA-256. Использование `expo-crypto.digest()` создало бы 200K асинхронных пересечений native bridge на React Native (Hermes), вызывая зависание приложения. Чистая JS-реализация выполняется синхронно в JS-потоке (~1-2 секунды).

**Почему NaCl secretbox вместо AES-GCM?** React Native на Hermes не имеет `crypto.subtle`. NaCl secretbox (XSalsa20-Poly1305) обеспечивает эквивалентное аутентифицированное шифрование через tweetnacl (чистый JS, без нативных зависимостей).

### Иерархия ключей

```
Парольная фраза шифрования (НЕ пароль для входа)
     | PBKDF2 (случайная соль, 100K итераций)
     v
Master Key (MK) -- 256 бит, никогда не покидает память устройства
     |
     |---> Оборачивает Account Key 1 (AK1) -- симметричный ключ для каждого аккаунта
     |       |---> Шифрует все данные Аккаунта 1
     |
     |---> Оборачивает Account Key 2 (AK2)
     |       |---> Шифрует все данные Аккаунта 2
     |
     |---> Оборачивает Identity Key Pair (X25519 + Ed25519)
             |---> Приватные ключи для получения AK от владельца общего аккаунта
             |---> Публичные ключи публикуются на сервере
```

### Хранение ключей

| Платформа | Хранилище | Безопасность |
|-----------|-----------|-------------|
| iOS | Keychain через expo-secure-store | Аппаратная защита |
| Android | EncryptedSharedPreferences через expo-secure-store | Аппаратная защита |
| Web | localStorage + PBKDF2 из парольной фразы | Парольная фраза каждую сессию |

Ключи аккаунтов кешируются локально в SQLite (таблица `encryption_keys`), обёрнутые Master Key. Они доступны только после разблокировки парольной фразой.

### Формат зашифрованных данных

Каждая сущность содержит колонку `encryptedPayload` (TEXT) с JSON-блобом:

```json
{
  "v": 1,
  "kv": 1,
  "fields": {
    "description": { "iv": "base64...", "ct": "base64...", "tag": "base64..." },
    "notes": { "iv": "base64...", "ct": "base64...", "tag": "base64..." }
  }
}
```

- `v` -- версия формата шифрования
- `kv` -- версия ключа аккаунта (для ротации ключей)
- Каждое поле имеет свой случайный 24-байтный nonce (`iv`), шифротекст (`ct`) и 16-байтный тег Poly1305 (`tag`)
- Оригинальные plaintext-колонки устанавливаются в `NULL` (текст) или `0` (числа) при шифровании

### Шифруемые поля по сущностям

| Сущность | Уровень 1 (текстовые поля) | Уровень 2 (дополнительно) |
|----------|---------------------------|--------------------------|
| Расход (Expense) | description, notes, locationName, receiptUrl | amount, discountAmount |
| Позиция расхода (ExpenseItem) | description | unitPrice, totalPrice |
| Доход (Income) | description, notes | amount |
| Бюджет (Budget) | name | amount |
| Категория (Category) | name | -- |
| Тег (Tag) | name | -- |
| Проект (Project) | name, description | budget |
| Обмен валют (CurrencyExchange) | notes | fromAmount, toAmount, exchangeRate |
| Разделение (ExpenseCategorySplit) | notes | amount, percentage |
| Баланс кошелька (WalletBalance) | -- | initialAmount |

#### Схема базы данных

**PostgreSQL (сервер)**

```prisma
model UserEncryptionProfile {
  id                         String   @id @default(uuid())
  userId                     String   @unique
  pbkdf2Salt                 String   // Base64-закодированная 32-байтная соль
  publicKeyX25519            String   // Base64-закодированный публичный ключ
  publicKeyEd25519           String   // Base64-закодированный публичный ключ
  wrappedPrivateKeyX25519    String   // Приватный ключ, зашифрованный MK
  wrappedPrivateKeyEd25519   String   // Приватный ключ, зашифрованный MK
  recoveryKeyHash            String?  // bcrypt-хеш ключа восстановления
  wrappedMasterKeyByRecovery String?  // MK, зашифрованный ключом восстановления
  keyVersion                 Int      @default(1)
  createdAt                  DateTime @default(now())
  updatedAt                  DateTime @updatedAt
}

model AccountEncryptionKey {
  id                String   @id @default(uuid())
  accountId         String
  userId            String
  wrappedAccountKey String   // AK, зашифрованный MK или ECDH-секретом
  wrappedBy         String   // userId того, кто обернул
  wrappingMethod    String   // 'master_key' | 'ecdh'
  keyVersion        Int      @default(1)
  @@unique([accountId, userId])
}

// Добавлено в модель Account:
// encryptionEnabled  Boolean @default(false)
// encryptionTier     Int     @default(0)   // 0=выкл, 1=текст, 2=полное
// keyRotationNeeded  Boolean @default(false)

// Добавлено во все синхронизируемые модели (Expense, Income, Budget, Category, Tag, Project и т.д.):
// encryptedPayload      String?
// encryptionKeyVersion  Int?
```

**SQLite (мобильное, локальный кеш)**

```sql
CREATE TABLE IF NOT EXISTS encryption_keys (
  account_id  TEXT PRIMARY KEY,
  account_key TEXT NOT NULL,    -- AK, обёрнутый MK (Base64)
  key_version INTEGER NOT NULL DEFAULT 1,
  updated_at  INTEGER NOT NULL
);
```

### Ротация ключей

Запускается при удалении участника из общего аккаунта или подозрении на компрометацию ключа.

1. Владелец генерирует новый AK
2. Оборачивает новый AK для каждого оставшегося участника через ECDH
3. Загружает все обёрнутые ключи на сервер
4. Сервер увеличивает `keyVersion`, снимает флаг `keyRotationNeeded`
5. Перешифрование существующих данных новым AK -- планируется как фоновый процесс

### Механизм восстановления

1. При настройке генерируется 256-битный ключ восстановления (отображается как `XXXX-XXXX-...`)
2. `AES-wrap(recovery_key, MK) --> wrapped_MK` сохраняется на сервере
3. `bcrypt(recovery_key)` сохраняется на сервере для верификации
4. Поток восстановления: пользователь вводит ключ --> сервер проверяет через bcrypt --> возвращает `wrapped_MK` --> клиент развёртывает MK --> пользователь устанавливает новую парольную фразу --> переоборачивает всё

### Свойства безопасности

**Что сервер никогда не видит:**
- Парольную фразу шифрования
- Master Key (MK) в открытом виде
- Account Keys (AK) в открытом виде
- Расшифрованные значения полей (описания, заметки, названия, суммы в уровне 2)
- Ключ восстановления в открытом виде (только bcrypt-хеш)

**Что сервер хранит:**
- Соль PBKDF2 (нужна для деривации ключей на новых устройствах)
- Публичные ключи (X25519, Ed25519)
- Обёрнутые (зашифрованные) приватные ключи
- Обёрнутые (зашифрованные) ключи аккаунтов
- Зашифрованные блобы полей
- bcrypt-хеш ключа восстановления + MK, обёрнутый ключом восстановления

### Конфигурация

`packages/shared-utils/src/constants/index.ts`:

```typescript
export const ENCRYPTION_CONFIG = {
  pbkdf2Iterations: 100000,      // OWASP рекомендует 600K+; 100K -- компромисс для мобильных
  aesKeyLengthBits: 256,         // Длина ключа NaCl secretbox
  ivLengthBytes: 12,             // Не используется (NaCl использует 24-байтный nonce внутри)
  saltLengthBytes: 32,           // Соль PBKDF2
  encryptionFormatVersion: 1,    // Для будущих миграций формата
  recoveryKeyLengthBytes: 32,    // 256-битный ключ восстановления
  migrationBatchSize: 50,        // Для будущего фонового перешифрования данных
} as const;
```

### Зависимости

- `tweetnacl` (v1.0.3) -- NaCl крипто: secretbox, box (X25519), sign (Ed25519)
- `tweetnacl-util` (v0.15.1) -- Помощники кодирования Base64/UTF-8
- `expo-crypto` -- Случайные байты на нативе (Hermes не имеет `crypto.getRandomValues`)
- `expo-secure-store` -- Доступ к Keychain/Keystore для хранения флага `e2ee_setup`
- `bcrypt` (только сервер) -- Хеширование ключей восстановления

### Тестирование

Файл тестов: `apps/mobile/src/services/__tests__/crypto.test.ts`

- Корректность SHA-256: проверена на известных тестовых векторах (пустая строка, "abc")

- Обёртка ключей: обёртка/развёртка round-trip
