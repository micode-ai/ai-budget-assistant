# Open Banking — что это, как работает и где действует

> Обзорный документ (исследование, июнь 2026). Не является техническим планом —
> план внедрения в AI Budget Assistant: `docs/superpowers/plans/2026-06-12-open-banking-integration.md`.

## 1. Что такое Open Banking

**Open Banking (открытый банкинг)** — это модель, при которой банки обязаны (или добровольно соглашаются) предоставлять лицензированным третьим сторонам программный доступ к данным счетов клиента и к инициированию платежей — **через стандартизированные API и только с явного согласия клиента**.

До Open Banking агрегаторы финансов работали через **screen scraping**: пользователь отдавал приложению свой банковский логин/пароль, и робот «заходил» в интернет-банк от его имени. Это небезопасно (передача учётных данных третьей стороне) и нестабильно (ломается при каждом редизайне банка). Open Banking заменяет это официальным каналом: согласие выдаётся в самом банке, приложение получает токен с ограниченными правами, пароль от банка никуда не передаётся.

Ключевые свойства:

- **Согласие (consent)** — клиент явно выбирает, какому приложению, к каким счетам и на какой срок дать доступ. Согласие можно отозвать в любой момент.
- **Только чтение или только платёж** — доступ разделён на роли (см. ниже); приложение для учёта финансов получает доступ *только на чтение*.
- **Регулируемость** — доступ к данным имеют только лицензированные/зарегистрированные провайдеры, за которыми надзирает финансовый регулятор.

## 2. Роли и термины

| Термин | Расшифровка | Кто это |
|---|---|---|
| **ASPSP** | Account Servicing Payment Service Provider | Банк, который ведёт счёт клиента |
| **TPP** | Third Party Provider | Третья сторона, получающая доступ (финтех-приложение, агрегатор) |
| **AISP** | Account Information Service Provider | TPP с правом **читать** данные счетов (история транзакций, балансы). Это роль PFM-приложений вроде нашего |
| **PISP** | Payment Initiation Service Provider | TPP с правом **инициировать платежи** со счёта клиента |
| **PSU** | Payment Service User | Конечный пользователь (клиент банка) |
| **SCA** | Strong Customer Authentication | Двухфакторная аутентификация, которую банк требует при выдаче согласия |
| **AIS-агрегатор** | — | Компания-посредник (Tink, Salt Edge, Enable Banking…), которая держит интеграции с сотнями банков и даёт один API вместо сотен |

## 3. Как это работает (поток AIS — чтение данных)

```
Пользователь            Приложение (мы)         Агрегатор / TPP            Банк (ASPSP)
    │  «Подключить банк»      │                        │                        │
    │ ───────────────────────►│  start consent         │                        │
    │                         │ ──────────────────────►│  создать сессию        │
    │                         │                        │ ──────────────────────►│
    │                  redirect на страницу банка (authUrl)                     │
    │ ◄──────────────────────────────────────────────────────────────────────── │
    │  логин в СВОЁМ банке + SCA (BLIK/SMS/биометрия) + выбор счетов            │
    │ ────────────────────────────────────────────────────────────────────────► │
    │                  redirect обратно (callback + code)                       │
    │                         │ ◄──────────────────────│ ◄───────────────────── │
    │                         │  токен доступа (только чтение, срок ~180 дней)  │
    │                         │                        │                        │
    │                         │  fetch transactions    │   GET /accounts,       │
    │                         │ ──────────────────────►│   GET /transactions    │
    │                         │ ◄──────────────────────│ ◄───────────────────── │
```

1. Пользователь в приложении выбирает свой банк.
2. Приложение (через агрегатора) создаёт запрос согласия и получает `authUrl`.
3. Пользователь уходит на сайт/в приложение **своего банка**, проходит SCA и подтверждает: «даю приложению X доступ на чтение к счетам Y на N дней».
4. Банк возвращает пользователя обратно (redirect/deep link), TPP получает токен.
5. Дальше приложение периодически забирает транзакции и балансы по API — без участия пользователя.

### Технические ограничения (PSD2, EU)

- **Срок согласия — до 180 дней** (изначально было 90; продлено поправкой EBA RTS). После истечения пользователь должен пройти SCA в банке заново («reconnect»).
- **Без участия пользователя — максимум 4 обращения в сутки** на счёт (фоновая синхронизация). Если пользователь активно в приложении — лимита нет.
- Доступ только к **платёжным счетам** (текущие счета, часто карты). Депозиты, брокерские счета, кредиты PSD2 не покрывает — это придёт с FIDA (см. ниже).
- История транзакций обычно ограничена тем, что отдаёт банк (часто 90 дней без SCA, больше — при первом подключении).

### Стандарты API

Единого мирового стандарта нет; основные семейства:

- **Berlin Group NextGenPSD2** — де-факто стандарт континентальной Европы (большинство банков EU).
- **UK Open Banking Standard** — Великобритания (OBL, бывш. OBIE), самый зрелый и единообразный.
- **STET** — Франция.
- **PolishAPI** — Польша (стандарт Związek Banków Polskich; локальный профиль поверх PSD2).
- **FDX** — Северная Америка (индустриальный, не регуляторный).

Из-за этого зоопарка почти никто не интегрируется с банками напрямую — используют **агрегаторов**.

## 4. Агрегаторы (как реально подключаются приложения)

| Провайдер | Покрытие | Особенности (на июнь 2026) |
|---|---|---|
| **Tink** (Visa) | ~3400 банков EU+UK | Энтерпрайз, sales-driven, дорого для малых команд |
| **Plaid** | США/Канада + часть EU/UK | Лидер в США; в Польше покрытие частичное |
| **TrueLayer** | UK + EU | Силён в UK, фокус на платежах (PIS) |
| **Salt Edge** | 5000+ банков, 50+ стран | Есть модель **delegated/under-our-licence** — можно работать агентом под их AISP-лицензией; цены по запросу |
| **Enable Banking** | EU/EEA + UK | Self-serve sandbox, объёмное ценообразование (по числу подключённых счетов), минимальный месячный счёт; подходит малым продуктам |
| **GoCardless Bank Account Data** (бывш. Nordigen) | EU + UK | Исторически бесплатный AIS-tier (50 подключений/мес), **но с осени 2025 закрыт для новых клиентов** — на него рассчитывать нельзя |
| **Yapily** | EU + UK | API-first, средний сегмент |
| **Kontomatik** | Польша + CEE | Польский KIP/AISP, глубокое покрытие локальных банков |

**Лицензионный вопрос.** Чтобы *самостоятельно* читать банковские данные, нужна регистрация AISP у регулятора (в Польше — KNF; без требования к капиталу, но со страховкой профответственности; процесс 6–12 месяцев, действует во всём EEA через паспортизацию). Малые продукты обычно идут другим путём — **работают агентом под лицензией агрегатора** (Salt Edge delegated model, аналогичные схемы у других) либо используют архитектуру, где токены и данные технически проходят через лицензированного провайдера. Это снимает основной регуляторный барьер.

## 5. В каких странах действует

### Зрелые регуляторные режимы (доступ обязателен для банков)

| Страна/регион | Режим | С какого года | Статус |
|---|---|---|---|
| **EU/EEA** (вкл. Польшу) | PSD2 | 2018–2019 | Работает повсеместно; качество API у банков неровное |
| **Великобритания** | Open Banking (CMA Order + PSRs) | 2018 | Самый зрелый рынок, >10 млн активных пользователей |
| **Австралия** | CDR (Consumer Data Right) | 2020 | Шире банкинга: энергетика, телеком |
| **Бразилия** | Open Finance (BCB) | 2021 | Один из крупнейших по объёму; уже open finance, не только banking |
| **Индия** | Account Aggregator (RBI) | 2021 | Своя модель «агрегаторов согласий» |
| **Бахрейн, Саудовская Аравия, ОАЭ** | национальные фреймворки | 2019–2023 | Активно развиваются |

### Переходные / частичные

| Страна | Статус (июнь 2026) |
|---|---|
| **США** | Регуляторного принуждения долго не было — рынок построен на FDX + screen scraping (Plaid). Правило CFPB §1033 принято в окт. 2024, но его судьба пересматривается (суды, смена политики регулятора); внедрение буксует |
| **Канада** | Фреймворк consumer-driven banking в процессе внедрения (законодательство 2024–2026) |
| **Украина** | Закон принят, открытый банкинг официально запущен в авг. 2025 (НБУ) |
| **Россия** | Стандарты «Открытых API» ЦБ; пилоты; обязательность для крупнейших банков анонсирована, в международные агрегаторы не интегрирована |
| **Сингапур** | SGFinDex — добровольная государственная модель |
| **Япония** | Поправки к Banking Act — банки «должны стремиться» открыть API (мягкое регулирование) |

### Что дальше: PSD3 / PSR / FIDA (EU)

- **PSD3 + PSR**: политическое соглашение Совета и Парламента достигнуто 27.11.2025, финальные тексты — H1 2026, применение ожидается **~2027–2028** (переходный период ~18–21 мес.). Для AIS главное: единые требования к качеству банковских API, выделенные dashboards управления согласиями, упрощение жизни TPP.
- **FIDA (Financial Data Access)** — «open finance»: расширение доступа с платёжных счетов на **инвестиции, пенсии, страхование, ипотеки, кредиты**. Ещё в трилоге; реалистичный срок применения — **2029–2030**.

## 6. Что это даёт AI Budget Assistant

Сегодня импорт транзакций у нас ручной: CSV/PDF-выписки (mBank, PKO, Revolut, Wise, Erste, Alior + universal mapper). Open Banking превращает это в «подключил банк один раз — транзакции подтягиваются сами»:

- автоматическая ежедневная синхронизация расходов/доходов без выгрузки файлов;
- балансы счетов в реальном времени (кошельки/виджеты);
- более качественные данные для AI-фич (anomaly alerts, Fat Finder, инсайты) — полнота и свежесть;
- сильный платный дифференциатор (фича уровня Pro/Business).

Цена вопроса: договор с агрегатором (помесячная плата за подключённые счета), consent-UX (переподключение раз в 180 дней), GDPR-обязанности (DPIA, минимизация данных). Детали — в плане внедрения.
